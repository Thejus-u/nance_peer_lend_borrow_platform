
Peer Lend & Borrow Platform
1. Background
You are working on Nance, a personal finance platform that helps users track real bank transactions, understand spending, and improve financial health.
Nance already ingests credit card transactions from a user’s Gmail by parsing bank alert emails, matching them to linked cards, and syncing spend.
This assignment extends that pattern to savings and current bank accounts, and introduces a peer lend/borrow system with repayment tracking, Gmail transaction ingestion, account discovery, notifications, real-time updates, and AWS deployment.
______________________________________________________
2. Objective
Design and implement a solution that:
Lets a Nance user create a loan to another user by mobile number
Requires the borrower to accept before the loan becomes active
Supports partial repayments with an accurate running balance
Shows each user their lent and borrowed positions
Provides a family ledger with net balances among linked family members
Fetches savings/current account bank transactions from Gmail
Detects bank accounts present in Gmail but not yet connected in the app
Sends notifications when loans are created, accepted, rejected, repaid, settled, or overdue
Pushes real-time WebSocket updates when loan status changes
Deploys end-to-end on AWS with PgBouncer in front of RDS
The solution must include backend APIs.
A simple frontend or API client, such as a Postman/Bruno collection or minimal UI, that demonstrates the flows is required.

4. Functional Requirements
4.1 User & Identity
Each user is identified by:
Mobile number, unique
Name
Optional profile image
One or more connected Gmail accounts
You may assume authentication is already handled or implement a minimal login/register flow.
Users may already be on Nance or may not yet be onboarded.
If the borrower is not on Nance, the loan remains in pending_invite until they sign up and accept.
4.2 Loan Creation — Lend Flow
A lender can create a loan with:
Borrower mobile number
Amount in INR
Optional note
Optional due date
Optional source bank transaction reference
Rules:
Lender and borrower must be different users
Amount must be greater than 0
Loan starts in pending_acceptance
Lender can cancel only before acceptance
Idempotency-Key header must be supported on loan creation
4.3 Loan Acceptance — Borrow Flow
The borrower should see incoming loan requests and can:
Accept ? loan becomes active
Reject ? loan becomes rejected
Until acceptance, the loan should not appear in the borrower’s active obligations.
4.4 Repayments
Repayments should support:
Amount
Date
Optional note
Optional bank/UPI transaction reference from ingested email data
Rules:
Total repayments cannot exceed the original loan amount
Balance = principal - sum of repayments
When balance reaches 0, loan status becomes settled
Repayments must be atomic under concurrent requests
4.5 Family Links & Family Ledger
Users can mark another Nance user as a family member with:
Relationship label
Visibility preference
The family ledger should show:
Per-member total lent
Per-member total borrowed
Net position
Optional:
Consolidated “A owes B ?X” across direct loans
4.6 Savings / Current Account Email Ingestion
Build Gmail-based ingestion for savings and current account transaction alerts.
Include:
UPI debits and credits
Debit card transactions linked to savings/current accounts
NEFT / IMPS / RTGS transactions
ATM withdrawals
Account debit / credit alerts
Balance update alerts where transaction details are available
Whitelisted bank sender domains
Exclude:
Credit card spend alerts
EMI / loan statement emails
Marketing or promotional emails
Bank cross-sell emails
Transaction records must include:
Amount
Date & time
Merchant / narration
Direction: debit or credit
Bank name
Masked account number
Account type
Source = gmail
Raw email reference ID
4.7 Bank Account Discovery from Gmail
While scanning Gmail, the system should detect bank accounts appearing in emails that are not yet linked inside Nance.
A discovered account should be stored with:
Bank name
Account type
Masked account number
First seen timestamp
Number of supporting emails
Status: unlinked, dismissed, or linked
Expected behavior:
If the account is not linked in Nance, surface a prompt asking the user to connect it
User can dismiss the suggestion
User can initiate account linking
Once dismissed or linked, the same account should not be prompted again
Important:
Do not confuse credit card last-4 digits with bank account last-4 digits
Only linked savings/current accounts should be used for transaction reconciliation and loan repayment matching
4.8 Notifications
Trigger notifications for:
New loan request received
Loan accepted
Loan rejected
Repayment recorded
Loan fully settled
Overdue reminder
New bank account discovered from Gmail
Each notification must have a dedupe key.
4.9 Real-Time WebSocket Updates
When loan state changes, push an event to connected clients for both lender and borrower.
Supported events:
loan.created
loan.accepted
loan.rejected
loan.repayment_recorded
loan.settled
Requirements:
Authenticate WebSocket connections
A user must only receive events for loans they are party to
Document reconnect and missed-event strategy
4.10 Transaction Reconciliation
If a repayment includes a bank transaction reference or is suggested from ingested Gmail data, the system should attempt to match it against savings/current account transactions.
Matching should consider:
Amount
Date proximity
Direction
Counterparty or narration, when available
Store match confidence:
high
medium
low
Rules:
Flag unmatched repayments for manual review
Never auto-settle a loan without user confirmation
4.11 Audit Trail
Every important state change must be stored in an append-only event log.
Events include:
Loan created
Loan accepted
Loan rejected
Loan cancelled
Repayment recorded
Loan settled
Email parsed
Bank account discovered
Notification created
Each event should store:
Timestamp
Actor
Event type
Entity reference
Payload snapshot

6. Edge Cases to Handle
Your solution should reasonably handle:
Borrower not on Nance yet
Concurrent repayment requests on the same loan
User A cannot read User B’s loan by guessing the ID
Partial repayments that would exceed principal
Same Gmail alert parsed twice
Credit card email misclassified as savings/current account transaction
Savings account last-4 colliding with credit card last-4 in email text
Discovered account dismissed, then new emails arrive
Gmail token expired
Loans with no due date
WebSocket client offline during status change
Perfect handling of every edge case is not required. Critical cases should be handled, and assumptions should be documented.
